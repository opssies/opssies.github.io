[FR] Aborder l'interopérabilité pour le programme CaRE D1

Si vous êtes un industriel, prestataire en cybersécurité, cette page vous fournira les éléments et étapes clés pour être embarqué dans le programme sous l'angle de l'interopérabilité.

Etape 1 : Connaître le programme CaRE D1 

Consultez le site de l'Agence du Numérique en Santé (ANS) introduisant les quatre axes du programme national

Etape 2 : Connaître le dispositif OPSSIES

Regardez la vidéo introduisant le dispositif OPSSIES et les enjeux du chantier d'interopérabilité

Etape 3 : Préparer l'embarquement de votre structure

Contactez l'ANS et suivez les instructions fournies par email par l'équipe CaRE/Industriels. Cette étape n'est pas détaillable par le présent canal.

Etape 4 : Confirmer l'embarquement sur le volet Communication

4.1 Téléchargez l'instance illustrative CaRE D1

Voici quelques points d'attention généraux :

  • pour CaRE D1, ne pas utiliser le schéma général, mais valoriser l'instance illustrative CaRE D1
  • ne pas altérer la syntaxe, la désignation des objets, attributs au format json ainsi que les qualifiants
  • préserver les deux signes underscore ("__") préfixant les attributs text
  • préserver les listes introduites par les crochets
  • consulter la FAQ dédiée au profil CaRE D1 pour la composition des relevés d'audits

4.2 A l'aide d'un éditeur de texte brut ou de code (tel que Notepad++ ou Textpad), modifiez l'instance téléchargée en suivant précisément les instructions ci-dessous. Sous l'objet events, modifiez le bloc d'identification de la structure réalisant les audits d'exposition Internet (en pratique, le bloc indiquant votre structure), en indiquant son SIREN et ses raisons sociales longue et abrégée.

{
"events": {
"stakeholders": [
{
"stakeholderId": {
"stakeholderIdType": 4,
"__text": "specifiez_ici_le_SIREN"
},
"stakeholderNameTypes": [
{
"stakeholderNameType": 1,
"__text": "specifiez_ici_la_raison_sociale_abregee"
},
{
"stakeholderNameType": 2,
"__text": "specifiez_ici_la_raison_sociale_complete"
}
],
"stakeholderType": 3,
"stakeholderRoleType": 2
},

4.3 Déposez l'instance JSON ainsi modifiée (rappel de la convention de nommage du fichier : SOCIETE-D1-AAAAMMJJ-HHMMSS.json) sur la boîte configurée pour votre structure par la DNUM. Signalez ensuite à la DNUM que vous avez effectué un dépôt d'instance de test.

4.4 Attendez le retour du MSP (DNUM et DNS) pour poursuivre les opérations sur le volet Sémantique du relevé d'audit.

Etape 5 : Confirmer l'embarquement sur le volet Sémantique

5.1 Sur la base de l'instance de test modifiée et validée par le MSP (DNUM et DNS), sous l'objet events, modifiez le bloc d'identification de la structure commanditaire des audits, applicable à l'ensemble des événements d'audits (event)

Il peut s'agir ou bien d'un groupement ou bien d'un établissement de santé autonome.

Dans le cas d'un groupement (ex. : GHT), utilisez le code suivant (en modifiant les identifiants et raisons sociales) :

{
"stakeholderId": {
"stakeholderIdType": 3,
"__text": "TST-00"
},
"stakeholderNameTypes": [
{
"stakeholderNameType": 6,
"__text": "GHT TEST"
},
{
"stakeholderNameType": 2,
"__text": "GROUPEMENT HOSPITALIER DE TERRITOIRE TEST"
}
],
"stakeholderType": 2,
"stakeholderRoleType": 1
}

Dans le cas d'établissement autonome, utilisez le code suivant (en modifiant les identifiants et raisons sociales) :

{
"stakeholderId": {
"stakeholderIdType": 3,
"__text": "000000001"
},
"stakeholderNameTypes": [
{
"stakeholderNameType": 1,
"__text": "CH TEST"
},
{
"stakeholderNameType": 2,
"__text": "CENTRE HOSPITALIER TEST"
}
],
"stakeholderType": 2,
"stakeholderRoleType": 1
}

5.2 Sous l'objet event, modifiez le bloc d'identification de la structure auditée et maximisez-en l'identification, afin de faciliter, en aval de la télétransmission du relevé, l'association de la prestation d'audit et de la structure auditée.

Note importante : un rôle donné ne peut pas être attribué à plus d'un acteur dans un élément event. Par exemple, sous event, il ne peut pas y avoir plusieurs acteurs dont le rôle est introduit par le code 4, mais un seul (en pratique, une structure auditée).

"event": [
{
"stakeholders": [
{
"stakeholderId": [
{
"stakeholderIdType": 3,
"__text": "specifiez_ici_le_finess_juridique"
},

{
"stakeholderIdType": 6,
"__text": "specifiez_ici_le_trigramme_ght_si_applicable"
},

{
"stakeholderIdType": 2,
"__text": "specifiez_ici_le_finess_geographique"
},

{
"stakeholderIdType": 4,
"__text": "specifiez_ici_le_siren_si_disponible"
},

{
"stakeholderIdType": 5,
"__text": "specifiez_ici_le_siret_si_disponible"
},
],
"stakeholderNameTypes": [
{
"stakeholderNameType": 1,
"__text": "HOPITAL TEST"
},
{
"stakeholderNameType": 2,
"__text": "HOPITAL UNIVERSITAIRE TEST"
}
],
"stakeholderType": specifiez_ici_code_1_si_ES_code_2_si_GHT,
"stakeholderRoleType": 4
}
],

5.3 Sous l'objet event, modifiez le bloc dates, en précisant une seule date, celle correspondant à la réalisation effective de l'audit. Il est ici recommandé de retenir l'horodatage associé à l'achèvement technique de l'audit (fin du scan) et non celui associé à son initialisation.

Rappel important : préservez la liste, même si une seule date est retenue ici.

"dates": [
{
"dateType": 7,
"__text": "specifiez_ici_la_date_de_realisation_de_l_audit_au_format_AAAAMMJJ-HHMMSS"
}
],

5.4 Sous l'objet event, et sous l'objet scopes, modifiez les blocs de description des actifs audités. 

Rappel important : les contextes fonctionnels et techniques sont exclusivement à renseigner sous l'objet scopes, et non à la racine event pour le profil CaRE D1. La description des actifs est recommandée par l'Agence du Numérique en Santé, en particulier les adresses IP et les domaines.

5.5 Sous l'objet event, précisez le niveau d'appréciation (attribut level) selon les règles établies par le CERT-Santé.

"outcome": {
"level": x
},

5.6 Préservez les qualifiants décrivant finalement l'événement :

"tlp": "amber",
"eventAnalysisType": 5,
"eventType": 29

Note : vous retrouverez l'ensemble des qualifiants et scénarii des croisement des qualifiants dans la section "Définitions" de la documentation du schéma d'intéropérabilité. N'hésitez pas à consulter la FAQ du profil CaRE D1, régulièrement enrichie.

5.7 Déposez l'instance JSON ainsi modifiée sur la boîte configurée pour votre structure par la DNUM. Signalez ensuite à la DNUM que vous avez effectué un dépôt d'instance avec des données de production.

5.8 Attendez le retour du MSP (DNUM et DNS) pour basculer définitivement en mode production, et communiquer officiellement sur votre service de télétransmission de relevés CaRE D1 auprès de vos partenaires hospitaliers.
